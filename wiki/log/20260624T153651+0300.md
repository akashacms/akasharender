# 2026-06-24 15:36 - Correct concurrency findings in Performance Measurement Methodology

New measurement data revealed that the techsparx.com `config.mjs` had
`config.concurrency` set to **1**, not the code default of 3. The original
`perf.json` overlap ratio of ~1.0 therefore confirmed only that that run
was serial *by configuration* -- it did not test whether concurrency
helps. A follow-up experiment varying `config.concurrency` on the fast
machine showed `user/real` rising to ~1.06 at concurrency 5+ (saturating
there), i.e. in-process `fastq`/`async` does produce genuine but small
(~6%) multi-core overlap from the async I/O-wait tail, while the ~94%
CPU-bound work still serializes on one JS thread. A long search-cache TTL
added a small further gain.

This also relates to a correction made (in the repo-root
`ARCHITECTURE-performance-review.md`, outside the wiki) to the earlier
overstated claim that the Mahabhuta pipeline is "inherently synchronous":
the pipeline is async with many `await` yield points, but `await` yields
*concurrency* (interleaving during I/O waits), not *parallelism* of CPU
work; a `setImmediate` in `MahafuncArray.process` would not change the
CPU-bound wall clock. Parallel page renders require multiple OS execution
contexts (child processes preferred over worker threads, given the
process-global `sqdb`/config/plugin singletons).

The wiki page was updated to:

- Reframe the concurrency pitfall to distinguish concurrency from
  parallelism and to note the wrong-setting measurement error.
- Correct the `config.concurrency` parameter description (code default 3,
  but sites may set it lower; gain saturates ~5 on CPU-bound sites).
