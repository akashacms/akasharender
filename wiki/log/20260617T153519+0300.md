# 2026-06-17 15:35 - Updated promised.node.sqlite Migration Architecture with Dependency Decisions

The project owner reviewed the migration architecture document and made
dependency decisions. The architecture page was revised to record them and
to add a detailed design for the key-value store shim, derived from reading
the installed `sq3-kv-data-store` source and the external plugins that use it.

## Analysis

- **`sqlite-vec` and `sqlite-lembed`**: confirmed not used in normal
  operation; both are gated behind `AK_LEMBED_MODEL` /
  `AK_LEMBED_MODEL_NAME`. They are removed for now and re-added when vector
  search is enabled. The guarded code paths (`doCreateVecDocumentsTable`,
  the `insert-lembed`/`update-lembed` runs, `semanticSearchDocs`) remain in
  place but dormant so re-enabling is a small change.
- **`sqlite3-query-log`**: removed; the `AK_PROFILE` SQL-profiling block in
  `lib/sqdb.ts` is deleted. The other two profiling levels (document-stage
  and Mahabhuta-function timing) are unaffected.
- **`sq3-kv-data-store`**: replaced with a local adapter/shim. Inspection of
  the installed `sq3-kv-data-store@1.0.1` showed it (a) hard-checks
  `DB instanceof sqlite3.Database` — which a `DatabaseSync` fails, causing a
  silent fallback to a new throwaway `:memory:` `sqlite3.Database` — and
  (b) uses node-sqlite3 callback-style `run`/`all`, which `DatabaseSync`
  does not provide. Therefore the shim must be a local reimplementation of
  the `SQ3DataStore` public API backed by `promised.node.sqlite`, not an
  object passed into the existing package.
- Verified the external plugins `akashacms-affiliates` and
  `akashacms-embeddables` call `newSQ3DataStore(name)` synchronously and use
  `get`, `put`, `find`, and `findAll`. This drove the recommendation for
  lazy table initialization (to keep `newSQ3DataStore` synchronous) and the
  requirement to faithfully port the `selectors2where` query builder.

## Changes

- Recorded the four dependency decisions in the dependency table and added a
  new subsection 2.1 on the impact of removing the vector/embedding
  extensions.
- Rewrote section 5.3 (profiling) and 5.4 (KV store) to state the confirmed
  decisions and the technical reasons the existing KV package cannot accept
  a `DatabaseSync`.
- Added section 5.7 describing the local `KVDataStore` shim design (shared
  `AsyncDatabase`, lazy `CREATE TABLE IF NOT EXISTS`, ported
  `selectors2where`, JSON storage, full public surface).
- Updated the target-design diagram, the `lib/sqdb.ts` change list, the
  `package.json` change list, the migration phases, and the risks section.
- Added Sources and Related Pages entries for the KV package, the external
  plugins, and the Key-Value Store concept; bumped `last-updated`.

## Recommendations

- Confirm the `package.json` reference strategy for the unpublished
  `promised.node.sqlite` (local `file:` vs git) at implementation time.
- During implementation, add direct tests for the KV shim's `get`, `put`,
  `find`, and `findAll`, including the `selectors2where` operators, since
  the shim is a reimplementation rather than a thin pass-through.
