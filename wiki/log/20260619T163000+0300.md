# 2026-06-19 16:30 - Added Answer Page on Open Graph / Twitter Cards / Facebook Social Metadata

Created a new answer page describing Open Graph, Twitter Cards, and the
Facebook equivalent, and proposing an improved AkashaCMS implementation, in
response to the user's request. The user noted the feature is partly
implemented but unsatisfying, and wants a simple way to add frontmatter that
ends up in Open Graph / Twitter metadata.

To ground the answer accurately, the existing implementation in the sibling
`@akashacms/plugins-base` repo was read directly:
`layouts/ak_headermeta.html.njk`, `partials/ak_headermeta.html.ejs`,
`index.mjs` (`fixHeaderMeta` / `akheadermetatags`), and `test/index.mjs`.

Key findings captured:

- Open Graph is the core vocabulary; Facebook consumes OG (only `fb:app_id`/
  `fb:admins` and `article:*` are Facebook-specific); Twitter Cards mostly fall
  back to OG. OG/FB use `property=`, Twitter uses `name=`.
- The current `ak_headermeta` partial emits `og:*` tags with the WRONG
  attribute (`name=` instead of `property=`), is internally inconsistent
  (later `property=` tags), has NO Twitter Card support, weak `article:*`/
  Facebook support, a verbose per-property `if`-ladder, and latent
  `arg.pagetitle` bugs in `fixHeaderMeta`. The base-plugin tests assert the
  buggy `name=` form, encoding the bug.
- Proposed improvement: a frontmatter-driven `social:` block plus a generic
  `extra:` pass-through, normalized into a `{attr,key,value}` data table that a
  single template loop emits — fixing `property` vs `name` centrally, adding
  one escaping point, and removing the need to edit templates per property.
  Strong fallbacks (title/description/date/author/tags, plus existing image
  promotion) mean most pages need no extra frontmatter. Includes migration/
  compatibility notes (keep legacy `metaOG*` keys, fix the tests, fix
  `fixHeaderMeta`).

Used an `External-Sources` frontmatter field plus inline "external source"
notes for the sibling-repo files, since `Sources` in the page format refers to
files within this repo; the read-only sibling working copy is cited explicitly
and marked NEEDS VERIFICATION for external standards.
