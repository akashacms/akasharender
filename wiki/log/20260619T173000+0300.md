# 2026-06-19 17:30 - Added Answer Page on the Webmention Protocol

Created a new answer page describing Webmention — its purpose, markup,
protocol, and server-side software — in response to the user's follow-up after
the microformats answer referenced Webmention.

Key points captured:

- Webmention is a W3C Recommendation for decentralized cross-site
  reply/like/repost notifications; the IndieWeb's open successor to Pingback,
  and complementary to (not competing with) ActivityPub.
- Markup: one `rel="webmention"` link (HTTP `Link` header or `<link>`/`<a>`) to
  receive; microformats2 `u-in-reply-to`/`u-like-of`/`u-repost-of`/`u-bookmark-of`
  plus `h-card`/`e-content` to give sent mentions meaning; no special markup
  needed to send a plain mention.
- Protocol: sender discovers the endpoint, POSTs `source`+`target` as
  form-encoded; the receiver validates, then FETCHES the source and verifies it
  actually links to the target (the anti-spam core), parses with microformats2,
  stores, displays, and handles update/delete; plus SSRF/sanitization notes.
- Server software: a static site cannot receive on its own. Hosted services
  (Webmention.io to receive, webmention.app/Telegraph to send, Bridgy to bridge
  silos/Fediverse) or a self-hosted receiver (a PHP endpoint on shared hosting,
  reusing the Fediverse PHP-shim pattern). Static-site sweet spot: receive via a
  hosted endpoint and fetch + render mentions at build time.
- Mapped onto AkashaCMS: emit `rel="webmention"` in the head pipeline, emit
  response microformats from frontmatter, fetch mentions at build time (using
  the SQLite cache), send via a Node sender or webmention.app, optional
  self-hosted PHP receiver.

External standards/services/libraries were marked NEEDS VERIFICATION.
