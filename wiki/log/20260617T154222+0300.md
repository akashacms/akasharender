# 2026-06-17 15:42 - Revised KV-Store Strategy to Modify sq3-kv-data-store Instead of a Local Shim

The project owner rejected the local-reimplementation (shim) approach for the
key-value store and chose instead to modify the `sq3-kv-data-store` package so
it recognizes a `node:sqlite` `DatabaseSync` and acts appropriately. Via a
follow-up question, the owner selected the variant: rewrite the package to be
node:sqlite-only and drop `sqlite3` entirely (a breaking major version),
with AkashaRender passing `sqdb.inner` (the `DatabaseSync`) to the
constructor. The architecture page was revised to match.

## Analysis

- Confirmed the package's own README already lists this as a roadmap TODO:
  "Support this against the SQLITE3 due to be implemented in Node.js 24"
  (installed `sq3-kv-data-store/README.md`). The package repo is
  `https://github.com/robogeek/sqlite3-key-value-data-store`.
- The package source repo is not currently a sibling directory of
  AkashaRender, so its development location is a decision point; the updated
  version is unpublished and must be referenced via a local/git dependency
  until published — parallel to the `promised.node.sqlite` situation.
- The package's public API and `selectors2where` query-building logic are
  driver-agnostic and unchanged; only the constructor (driver detection /
  string-open), the query execution (callback-style `run`/`all` ->
  `StatementSync`), table creation (`IF NOT EXISTS`, now synchronous via
  `DatabaseSync.exec`), and dependencies (`sqlite3` removed) change.
- Because `DatabaseSync.exec` is synchronous, the constructor can finish
  table creation inline, preserving the synchronous `newSQ3DataStore(name)`
  signature external plugins rely on. This removes the earlier "async
  constructor / lazy init" concern from the shim design.

## Changes

- Dependency table: `sq3-kv-data-store` disposition changed from

## Recommendations

- Decide where the `sq3-kv-data-store` source repo will live locally for the
  companion update, and how AkashaRender's `package.json` references it

  `findAll` and the `selectors2where` operators under `node:sqlite` before
