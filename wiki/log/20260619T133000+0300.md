# 2026-06-19 13:30 - Expanded Fediverse Answer with PHP Shim and Entity Explanations

Revised the ActivityPub/Fediverse answer page in response to follow-up
questions: (1) whether WebFinger could be a simple PHP script and whether a
suitable one already exists, given that shared hosting typically provides PHP;
and (2) a request for fuller explanation of the Tier 3 entities — Actor,
Collections, and Objects/Activities — and where PHP scripts could fit.

## Changes

- Reframed the intro so the "static cannot federate" limitation is qualified by
  the shared-hosting reality that PHP is almost always available; the hard
  "cannot" now applies only to no-server-side hosts such as GitHub Pages.
- Added a "Shared hosting reality" subsection to Tier 2 explaining why static
  files cannot answer the WebFinger `?resource=` query or set the JRD
  content-type, with a minimal `webfinger` PHP script, an `.htaccess` rewrite,
  and a note that reusable PHP WebFinger snippets/libraries already exist
  (marked NEEDS VERIFICATION).
- Substantially expanded Tier 3 with plain-English explanations of **Actor**
  (the account/profile, with example JSON and the role of `publicKey`/HTTP
  Signatures), **Collections** (outbox/followers/following/inbox, noting which
  are static-friendly vs inherently dynamic), and **Objects/Activities**
  (Note/Article vs Create/Update/Delete/Announce/Like, addressing, example
  JSON). Added a subsection mapping the dynamic surface onto concrete PHP
  scripts (`inbox.php`, content-negotiating router, delivery script) plus the
  `landrok/activitypub` library and Bridgy Fed as a zero-server alternative.
- Updated the phased plan (Phases 2–4) to ship PHP templates alongside static
  artifacts, and updated the Bottom Line and the NEEDS VERIFICATION note.
- Updated frontmatter `last-updated` and added `webfinger` / `php-hosting`
  categories.
- Updated the summaries in `wiki/answers/README.md` and `wiki/index.md`.
