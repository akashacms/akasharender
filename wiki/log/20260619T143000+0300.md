# 2026-06-19 14:30 - Added Answer Page on Bluesky / AT Protocol Integration

Created a new answer page addressing whether there is crossover between the
Fediverse and Bluesky, whether Bluesky can query an AkashaCMS site, and how an
AkashaCMS website should present itself to Bluesky. Filed as a separate answer
document per the user request.

Key points captured:

- The Fediverse (ActivityPub) and Bluesky (AT Protocol / atproto) are separate
  networks that do not federate natively; crossover happens only through
  bridges such as Bridgy Fed, at the account level.
- Bluesky does not crawl websites for content (content lives in atproto
  repos/PDS). The two static-site-relevant fetches are Open Graph link-card
  metadata and an optional domain-handle verification artifact.
- An AkashaCMS site presents well to Bluesky by reusing the shared Open Graph
  metadata (already injected via `mahaMetadata` in the Built-in Plugin) and by
  optionally generating `/.well-known/atproto-did` (or documenting the
  `_atproto` DNS TXT record) so the domain can serve as the owner's handle.
- Recommended folding this into the same plugin proposed in the Fediverse
  answer, and deferring protocol-level cross-network presence to a bridge.

External atproto/Bluesky details were marked NEEDS VERIFICATION since they are
evolving web standards/services rather than AkashaRender source.
