# Vector Embeddings

## Definition

Vector Embeddings are numerical representations of text content as high-dimensional vectors (typically 384 floating-point values) generated by machine learning models, enabling semantic similarity comparisons between documents based on meaning rather than keyword overlap. AkashaRender optionally computes and stores embeddings for document body text in a specialized sqlite-vec virtual table, allowing nearest-neighbor searches to find semantically related content (source: [lib/cache/sql/create-table-vec-documents.sql](../../lib/cache/sql/create-table-vec-documents.sql), [lib/cache/sql/insert-lembed-documents.sql](../../lib/cache/sql/insert-lembed-documents.sql)).

## How It Works

Vector embeddings are generated and stored when the embedding model is configured (source: [lib/cache/cache-sqlite.ts](../../lib/cache/cache-sqlite.ts), [lib/sqdb.ts](../../lib/sqdb.ts)):

**Virtual Table Storage**: A specialized `vec_documents` table using sqlite-vec's `vec0` virtual table type stores embeddings (source: [lib/cache/sql/create-table-vec-documents.sql](../../lib/cache/sql/create-table-vec-documents.sql):3-8):
```sql
CREATE VIRTUAL TABLE IF NOT EXISTS vec_documents USING vec0(
    vpath TEXT,
    body_embeddings FLOAT[384]
);
```

**384-Dimensional Vectors**: Each embedding is an array of 384 floating-point numbers, the output dimension of common sentence-transformer models like `all-MiniLM-L6-v2` (source: [lib/cache/sql/create-table-vec-documents.sql](../../lib/cache/sql/create-table-vec-documents.sql):7).

**Embedding Generation**: When a document is cached, if `lembedModelName` is defined, the `lembed()` SQL function processes document body text through the ML model (source: [lib/cache/sql/insert-lembed-documents.sql](../../lib/cache/sql/insert-lembed-documents.sql):9, [lib/cache/cache-sqlite.ts](../../lib/cache/cache-sqlite.ts):1378-1390):
```sql
INSERT INTO vec_documents(vpath, body_embeddings)
VALUES ($vpath, lembed($lembedModel, $bodyEmbed));
```

**Model Integration**: The `lembed()` function is provided by the sqlite-lembed extension, which wraps an ONNX runtime to execute the ML model within SQLite (source: [lib/sqdb.ts](../../lib/sqdb.ts):32,63).

**Conditional Storage**: Embeddings are only computed and stored when both environment variables `AK_LEMBED_MODEL` and `AK_LEMBED_MODEL_NAME` are set, making vector search opt-in (source: [lib/cache/cache-sqlite.ts](../../lib/cache/cache-sqlite.ts):1378,1454, [lib/sqdb.ts](../../lib/sqdb.ts):49-54).

**Document Association**: Each embedding is linked to a document via its `vpath`, allowing lookups and joins with the main DOCUMENTS table (source: [lib/cache/sql/create-table-vec-documents.sql](../../lib/cache/sql/create-table-vec-documents.sql):4).

**Update on Change**: When documents are modified, embeddings are regenerated using the same SQL pattern (source: [lib/cache/cache-sqlite.ts](../../lib/cache/cache-sqlite.ts):1428,1454-1460).

**Title Embeddings Disabled**: The system has infrastructure for title embeddings but currently only uses body embeddings (source: [lib/cache/sql/create-table-vec-documents.sql](../../lib/cache/sql/create-table-vec-documents.sql):6, [lib/cache/sql/insert-lembed-documents.sql](../../lib/cache/sql/insert-lembed-documents.sql):8).

## Key Parameters

**FLOAT[384]**: Data type specification for vector columns, indicating 384-dimensional floating-point vectors (source: [lib/cache/sql/create-table-vec-documents.sql](../../lib/cache/sql/create-table-vec-documents.sql):7).

**$lembedModel**: SQL parameter holding the model name used with `lembed()` function (source: [lib/cache/sql/insert-lembed-documents.sql](../../lib/cache/sql/insert-lembed-documents.sql):9).

**$bodyEmbed**: SQL parameter containing the document body text to be embedded (source: [lib/cache/sql/insert-lembed-documents.sql](../../lib/cache/sql/insert-lembed-documents.sql):9).

**$vpath**: Document virtual path associating embedding with document (source: [lib/cache/sql/insert-lembed-documents.sql](../../lib/cache/sql/insert-lembed-documents.sql):7).

**lembedModelName**: TypeScript constant exported from sqdb.ts, used to check if embedding is enabled (source: [lib/cache/cache-sqlite.ts](../../lib/cache/cache-sqlite.ts):1378,1454).

**vec0**: sqlite-vec virtual table module for efficient vector storage and similarity search (source: [lib/cache/sql/create-table-vec-documents.sql](../../lib/cache/sql/create-table-vec-documents.sql):3).

## When To Use

**Semantic Search**: Enable embeddings when you need content-based search that understands meaning, not just keyword matches (source: [lib/cache/sql/create-table-vec-documents.sql](../../lib/cache/sql/create-table-vec-documents.sql)).

**Related Content**: Find documents similar to a given document based on content similarity rather than shared tags or categories (source: [lib/cache/sql/create-table-vec-documents.sql](../../lib/cache/sql/create-table-vec-documents.sql)).

**Question Answering**: Match user queries to relevant documents even when exact keywords don't appear (source: [lib/cache/sql/create-table-vec-documents.sql](../../lib/cache/sql/create-table-vec-documents.sql)).

**Content Recommendation**: Suggest articles based on semantic similarity to what users are currently reading (source: [lib/cache/sql/create-table-vec-documents.sql](../../lib/cache/sql/create-table-vec-documents.sql)).

**Clustering**: Group similar documents together based on vector similarity for organization or navigation features (source: [lib/cache/sql/create-table-vec-documents.sql](../../lib/cache/sql/create-table-vec-documents.sql)).

## Risks & Pitfalls

**Performance Cost**: Embedding generation is CPU-intensive. Each document requires ML model inference, significantly increasing build times (source: [lib/cache/cache-sqlite.ts](../../lib/cache/cache-sqlite.ts):1378-1390).

**Memory Requirements**: ML models require substantial RAM (100MB+ for model, plus working memory). In-memory database combined with embeddings can exhaust available memory on large sites (source: [lib/sqdb.ts](../../lib/sqdb.ts):56-73).

**Model Selection**: The 384-dimension size is hardcoded, limiting model choice to compatible architectures. Different models require code changes (source: [lib/cache/sql/create-table-vec-documents.sql](../../lib/cache/sql/create-table-vec-documents.sql):7).

**Storage Overhead**: Each embedding is 384 floats × 4 bytes = 1.5KB per document. Large sites with thousands of documents consume significant storage (source: [lib/cache/sql/create-table-vec-documents.sql](../../lib/cache/sql/create-table-vec-documents.sql):7).

**Configuration Complexity**: Requires downloading ML models, setting environment variables correctly, and ensuring sqlite-lembed/sqlite-vec extensions are available (source: [lib/sqdb.ts](../../lib/sqdb.ts):49-73).

**Limited to Body Text**: Currently only document body is embedded, not titles, summaries, or metadata. This may miss important semantic signals (source: [lib/cache/sql/create-table-vec-documents.sql](../../lib/cache/sql/create-table-vec-documents.sql):6-7).

**No Incremental Updates**: Changing the embedding model requires regenerating all embeddings. No migration or versioning system exists (source: [lib/cache/cache-sqlite.ts](../../lib/cache/cache-sqlite.ts)).

**Virtual Table Limitations**: vec0 virtual tables don't support all standard SQL operations. Query patterns must match what sqlite-vec provides (source: [lib/cache/sql/create-table-vec-documents.sql](../../lib/cache/sql/create-table-vec-documents.sql):3).

**Text Preprocessing**: No documented text cleaning or preprocessing before embedding. HTML tags, markdown syntax, etc. are embedded as-is, potentially affecting quality (source: [lib/cache/sql/insert-lembed-documents.sql](../../lib/cache/sql/insert-lembed-documents.sql):9).
